# Roadmap

Этот документ фиксирует планируемые изменения проекта.

Текущее реализованное состояние, обязательные runtime-сценарии и действующая архитектура описаны в [README.md](/Users/antonpancenko/Documents/academy/README.md).

## Webstudio Integration

### Цель

Расширить проект так, чтобы страницы, собранные в `Webstudio`, опубликованные через Docker CLI и размещенные на нашем сервере, могли получать публичные данные из `Strapi` по стабильному API без дублирования бизнес-логики на клиенте.

### Базовые принципы

- `Strapi` остается единственным backend и единственным источником данных.
- `Webstudio` используется только как слой публичного представления.
- Цены, скидки, derived date fields и публичный DTO курса продолжают вычисляться только на backend.
- Интеграция должна работать через один боевой домен и same-origin запросы.
- Из-за небольшого ожидаемого количества сущностей серверная пагинация в этот контур не добавляется.

### Этап 1. Зафиксировать целевую архитектуру

- Добавить `Webstudio` как поддерживаемую публичную поверхность проекта.
- Оставить `Strapi` backend-слоем для страницы расписания, Tilda и будущих страниц `Webstudio`.
- Зафиксировать схему маршрутизации через reverse proxy:
  - `/api/*`, `/admin/*`, `/uploads/*`, `/assets/*` идут в `strapi`
  - публичные страницы идут в `webstudio`
- Свести интеграцию к относительным запросам вида `fetch('/api/...')`, чтобы не зависеть от внутренних адресов контейнеров и не вводить CORS без необходимости.

### Этап 2. Выделить public API для Webstudio

- Добавить отдельный стабильный namespace для `Webstudio`, не завязанный на admin endpoints.
- Базовый контракт:
  - `GET /api/public/courses`
  - `GET /api/public/courses/:slug`
  - `GET /api/public/courses/resolve?path=/course-page`
- Переиспользовать текущую сериализацию курса и единый public DTO.
- Поддержать выбор полей через `fields=...`.
- Поддержать фильтрацию по базовым сценариям:
  - `search=...`
  - `waitlist=true|false`
  - при необходимости фильтры по статусу и дате
- Не добавлять пагинацию, пока количество сущностей остается небольшим и может безопасно обслуживаться одним ответом.

### Этап 3. Расширить backend без дублирования логики

- Вынести общую логику публичной выборки так, чтобы она переиспользовалась между Tilda и `Webstudio`.
- Не копировать бизнес-правила в новый middleware или helper.
- Сохранить различие между:
  - `/api/courses-feed` для встроенной страницы расписания
  - `/api/tilda/*` для Tilda
  - `/api/public/*` для `Webstudio`
- Добавить regression-проверки на:
  - выбор полей
  - фильтрацию
  - поиск курса по `slug` или `path`
  - корректный `404` для отсутствующего курса

### Этап 4. Сделать клиентский интеграционный слой для Webstudio

- Не размножать произвольные inline-скрипты по страницам.
- Сделать единый helper по аналогии с `assets/tilda-course-fields.js`.
- Нужны два сценария:
  - вывод одного курса
  - вывод списка курсов
- Поддержать декларативную конфигурацию через `data-*` атрибуты:
  - `data-api-base`
  - `data-course-slug`
  - `data-fields`
  - `data-filter-waitlist`
  - `data-course-path`
- Helper должен уметь:
  - загружать данные
  - выставлять predictable loading / empty / error states
  - подставлять значения в DOM
  - не дублировать backend-логику вычисления цен и дат

### Этап 5. Подготовить production-контур для Docker CLI deploy

- Перевести старый `webstudio`-сценарий из legacy-статуса в отдельный поддерживаемый deploy path.
- Поднять production-связку:
  - `strapi`
  - `webstudio`
  - `caddy` или `nginx`
- Наружу публиковать только `80/443`.
- TLS и домен обслуживать на reverse proxy, а не на уровне отдельных контейнеров.
- Описать обязательные env-переменные и команды запуска для этого сценария.

### Этап 6. Зафиксировать SEO-правила для Webstudio-страниц

- Для SEO-важных страниц основной контент должен быть доступен в HTML без ожидания сложного клиентского рендера.
- API использовать для интерактивных и повторно используемых блоков, а не как единственный способ показать основной контент страницы.
- Для каждой публичной страницы обеспечить:
  - уникальный `title`
  - `meta description`
  - `canonical`
  - Open Graph мета
  - корректные `200`, `301`, `404`
  - `robots.txt`
  - `sitemap.xml`
- Определить единый источник SEO-метаданных для страниц курсов, чтобы не было расхождения между `Strapi` и `Webstudio`.

### Этап 7. Усилить эксплуатацию и проверки

- Добавить smoke-check для `Webstudio`-сценария:
  - главная публичная страница
  - `GET /api/health/ready`
  - `GET /api/public/courses`
  - `GET /api/public/courses/{slug}`
- Добавить rollback-инструкцию для `webstudio`-контейнера.
- Логировать ошибки public API отдельно от admin-сценариев.
- Ограничить public API только разрешенными полями и только опубликованными сущностями.

### Этап 8. Обновить проектную документацию

- Обновить [README.md](/Users/antonpancenko/Documents/academy/README.md) после реализации, когда `Webstudio` перестанет быть legacy-only сценарием.
- Добавить runbook по публикации `Webstudio`-контейнера на сервере.
- Добавить пример интеграции данных курса на странице `Webstudio`.
- Зафиксировать production flow, проверки после деплоя и критерии rollback.

## Критерии готовности

- Страницы `Webstudio` получают данные курсов через стабильный `/api/public/*` контракт.
- Все публичные поля продолжают формироваться на backend.
- Публичный домен обслуживается через reverse proxy с корректной маршрутизацией в `strapi` и `webstudio`.
- SEO-значимый контент страницы доступен без зависимости от поздней клиентской подгрузки.
- Документация и smoke-check покрывают новый сценарий deployment и эксплуатации.
